<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| reviewer output → parser | LLM/CLI text is untrusted; parsed via existing parse_findings (fence-strip, json-repair, per-finding schema validation). Unchanged by this phase. |
| git subprocess → run metadata | `git rev-parse`/`git status` output read into the run block; wrapped so a failure degrades to commit=None/dirty=None, never aborts or injects. |
| CLI args → filesystem | `--output` path controls where the JSON artifact is written (local-only tooling, operator-supplied). |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-31-01 | Denial of Service | ThreadPoolExecutor fan-out | mitigate | Bounded pool, default --max-workers 6; each unit wrapped in try/except so one hung/raising unit cannot abort the run (RUN-02). |
| T-31-02 | Tampering | git metadata seam | mitigate | git_info collected via an injectable seam wrapped so subprocess failure degrades to commit=None/dirty=None; tests inject a fake, no real-git dependency (RUN-03). |
| T-31-03 | Information Disclosure | --output artifact write | accept | Local standalone tooling under scripts/; operator supplies the output path; no secrets in the findings artifact (findings are file:line references + severities). |
| T-31-04 | Tampering | reviewer (LLM) output | accept | Untrusted text already constrained by the Phase 30 parse_findings schema validator; this phase only aggregates already-validated findings — no new parse surface. |
| T-31-SC | Tampering | npm/pip/cargo installs | mitigate | No new package installs in this phase (stdlib only: concurrent.futures, json, subprocess, datetime); slopcheck N/A. If any install is later proposed, gate behind a blocking-human checkpoint. |
</threat_model>
