# v1.1 Retrospective

## Outcome

v1.1 shipped on 2026-05-11 after 7 phases and 12 plans. The milestone closed the WUI/TUI parity
gap, added operator-controlled update and feedback paths, and introduced Jira-driven work intake
with classification, history refresh, repo hints, and code/test readiness gates.

## What Worked

- Keeping operator UI metadata in one shared contract made TUI/WUI parity testable instead of
  relying on duplicated template and TUI literals.
- Treating update checks and feedback as inserted urgent phases kept operator-adoption work inside
  the same milestone without hiding it in unrelated implementation commits.
- Reusing existing Jira, release-context, repo-target, and Postgres patterns avoided a parallel
  issue-workflow model.
- Focused tests kept milestone verification fast while still covering route drift, stale selectors,
  comment commands, Jira refresh hashes, repo hints, and readiness gates.

## What To Improve

- Live Jira/GitLab authentication should be smoke-tested on an operator machine before claiming
  production readiness for external provider usage.
- Docker-backed Alembic full-chain verification should run before migration-heavy future releases.
- `whilly jira poll` is intentionally one-shot; a future milestone should decide whether Whilly owns
  a long-running watch loop or documents scheduler-based operation as the supported mode.
- Nyquist `VALIDATION.md` artifacts were not generated for inserted phases. Future urgent inserted
  phases should either generate validation artifacts or explicitly record why focused verification
  is sufficient.

## Follow-Up Candidates

- Real-machine Jira/GitLab/OpenCode onboarding smoke.
- Jira watch daemon or scheduler wrapper.
- Browser and assistive-technology QA for the full WUI operator workflow.
- Expanded provider smoke tests for Docker, Colima, and local model/runtime availability.
