The planner already read the source. These findings are LOAD-BEARING — verify them
yourself when you read the modules, then spec accordingly:

SAFE-01 (resource_monitor.py + smoke.py):
- `ResourceMonitor` is a STANDALONE library with ZERO callers in whilly/ (grep
  confirmed unwired into the v4 worker-claim path). Spec what it DOES, and state it
  is not currently wired into the orchestration loop.
- `ResourceLimits` defaults: max_cpu_percent=80, max_process_cpu=50,
  max_memory_percent=75, max_process_memory_mb=2048, min_free_space_gb=5,
  max_log_dir_size_gb=2, max_concurrent_processes=5, process_timeout_minutes=30,
  monitor_interval_seconds=10.
- `check_limits` returns a violations dict with per-resource severity (high if cpu>90 /
  mem>90 / disk<1GB, else medium; log_dir is low). `should_throttle` returns True on
  any high-severity violation OR ≥2 medium-severity violations. `wait_for_resources`
  polls until under limits or max_wait (default 300s), throttled warnings every 60s.
- Env overrides via create_monitor_from_env: WHILLY_MAX_CPU_PERCENT,
  WHILLY_MAX_MEMORY_PERCENT, WHILLY_MIN_FREE_SPACE_GB, WHILLY_PROCESS_TIMEOUT_MINUTES.
- BUDGET in v4 is NOT the v3 kill-tmux→exit-2 path. The real budget contract is a
  Postgres audit sentinel: whilly/adapters/db/repository.py defines
  BUDGET_EXCEEDED_EVENT_TYPE = "plan.budget_exceeded", BUDGET_EXCEEDED_REASON =
  "budget_threshold", BUDGET_EXCEEDED_THRESHOLD_PCT = 100, emitted exactly once when
  plans.spent_usd transitions from < budget_usd to >= budget_usd (budget_usd=0 means
  unlimited per config.py). Spec THIS, not the v3 lore.
- smoke.py: shared smoke-report foundation with exit codes EXIT_OK=0,
  EXIT_CHECK_FAILED=1, EXIT_CONFIG_MISSING=2; SmokeReport.all_passed; _redact_url
  strips user:pass@ from URLs; write_smoke_report writes timestamped secret-free JSON.
- For process exit codes generally, REFERENCE the cli-surface spec contract
  (EXIT_OK=0/EXIT_VALIDATION_ERROR=1/EXIT_ENVIRONMENT_ERROR=2) rather than re-asserting
  a v3 budget→exit-2 path.

SAFE-02 (recovery.py + self_healing.py):
- BOTH modules are UNWIRED (grep confirmed zero callers in whilly/). recovery.py
  operates on the LEGACY in-memory TaskManager + progress.txt / whilly_logs/*.log
  files. self_healing.py installs a global sys.excepthook. Spec what they DO and mark
  them LEGACY, the way state-persistence/spec.md marks the v3 StateStore legacy.
- recovery.recover_task_statuses: reads done IDs from progress.txt (lines like
  "[id] DONE") and from per-task .log files containing COMPLETION_MARKER (parsing
  {"type":"result"} JSON lines, is_error False), unions them, flips matching non-done
  tasks to "done" on the legacy TaskManager, saves only if changes occurred, returns a
  {task_id: "old → done"} change dict.
- recovery.validate_task_consistency: returns warning strings for tasks completed
  per files but not marked done, and tasks marked done with no completion evidence.
- self_healing.SelfHealingHandler.analyze_error: regex-matches NameError /
  missing-positional TypeError / Import(ModuleNotFound)Error / AttributeError out of a
  traceback, returns a CodeError with a suggested_fix. apply_fix only implements
  NameError (logs suggestion, returns True) and ImportError (pip-installs the module).
  global_exception_handler runs analyze→apply then prints the full traceback;
  enable_self_healing sets sys.excepthook.
- The v4 LIVE equivalent of crash recovery is release_stale_tasks (visibility-timeout
  sweep + optimistic locking) documented in state-persistence/spec.md. Reference it.

Coverage matrix (Phase 28) will check these modules are accounted for — do not add
unrelated modules.
</grounding_findings>
