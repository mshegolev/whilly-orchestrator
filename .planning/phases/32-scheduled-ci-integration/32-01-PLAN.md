<objective>
Wire the Phase 31 fleet runner into a SCHEDULED CI job that runs the semantic
spec-drift check unattended, uploads the JSON artifact, renders the human
summary into the run, and applies a configurable gating posture — without
touching the v1.4 per-PR mechanical gate.

Two pieces:
1. CI-02 — add a `--fail-on {none,high}` gating arg to
   `scripts/semantic_drift_check.py`. Default `none` keeps every run report-only
   (exit 0). `--all --fail-on high` exits 1 iff any finding has severity `HIGH`;
   per-unit `errors` never gate; the `--slug` path is unchanged (still exit 0).
2. CI-01 — add a NEW `.github/workflows/semantic-drift.yml` triggered ONLY by
   `schedule` (weekly cron) + `workflow_dispatch`, decoupled from `ci.yml`. It
   installs the Claude CLI, fails fast if `ANTHROPIC_API_KEY` is absent, runs the
   fleet check, uploads the artifact, and writes the summary to the step summary.
   A `workflow_dispatch` `posture` input (report-only | fail-on-high) maps to
   `--fail-on none | high` via an allowlist + env, never inlined into the shell.

Purpose: make the semantic guard run repeatably on a bounded cadence with a
human-visible result and an operator-selectable gate, while the mechanical gate
in `ci.yml` stays the per-PR blocker untouched.

Output: gated script + tests, scheduled workflow + structural test, usage docs
for the required secret.

Scope guard: `scripts/` + `.github/` + `tests/` + docs ONLY — zero `whilly/`
change, so no `opsx` spec delta obligation. Do NOT build the known-drift fixture
(Phase 33).
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| workflow_dispatch input → CI shell | An operator-supplied `posture` string crosses into the runner; untrusted until allowlisted. |
| GitHub secrets → CI env | ANTHROPIC_API_KEY enters the job env; must not leak into logs or artifacts. |
| npm registry → CI runner | The Claude CLI package install is a supply-chain edge. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-32-01 | Tampering | `posture` workflow_dispatch input inlined into `run:` shell | mitigate | Validate against the {report-only, fail-on-high} allowlist; pass via a quoted `POSTURE` env var mapped through a case/if to `--fail-on`; never interpolate `${{ github.event.inputs.posture }}` into a shell line. Structural test asserts no inline interpolation. |
| T-32-02 | Information Disclosure | ANTHROPIC_API_KEY in CI | mitigate | Inject only via step `env:` from `secrets.ANTHROPIC_API_KEY`; never echo it; fail-fast step checks emptiness without printing the value. |
| T-32-03 | Denial of Service (false signal) | Flaky LLM run reds the gate | accept | Default posture report-only (exit 0) on cron; fail-on-high is opt-in; per-unit errors never gate (only HIGH findings do). Weekly cadence bounds cost. |
| T-32-04 | Spoofing | Silent pass when the key is absent | mitigate | Dedicated fail-fast step exits 1 with a clear message if ANTHROPIC_API_KEY is empty — a check that did not run never reports green. |
| T-32-SC | Tampering | npm install -g @anthropic-ai/claude-code | accept | First-party Anthropic-published package; install scoped to the ephemeral runner; no `whilly/` dependency change. No new pip/cargo install in this phase, so no Package Legitimacy Gate table is required. |
</threat_model>
