# Phase 13.2: GitHub Feedback Issue Reporter - Context

**Gathered:** 2026-05-11
**Status:** Ready for planning

<domain>
## Phase Boundary

Add one fast feedback channel so an operator trying Whilly on another computer can file bugs or
ideas directly into GitHub Issues. This phase intentionally does not add email, GitLab, Jira, or
automatic crash reporting.

</domain>

<decisions>
## Implementation Decisions

### Feedback Channel
- Use GitHub Issues through the existing `gh` CLI.
- Add a top-level `whilly feedback` command rather than hiding feedback inside unrelated flows.
- Support `bug` and `idea` reports with default labels.
- Keep `--dry-run` so operators can inspect the generated command before creating an issue.

### Safety
- Redact known secret patterns from the report body.
- Use `gh_subprocess_env()` for GitHub auth discipline.
- Do not send reports automatically.

### Claude's Discretion
- Default repository is `mshegolev/whilly-orchestrator`, overrideable with `WHILLY_FEEDBACK_REPO`
  or `--repo`.

</decisions>

<code_context>
## Existing Code Insights

### Reusable Assets
- `whilly.gh_utils.gh_subprocess_env()` centralizes GitHub CLI auth.
- `whilly.security.secret_lint.redact_secrets()` already redacts token-like content.
- `whilly.cli.__init__.main()` lazily dispatches top-level commands.

### Established Patterns
- CLI commands return integer exit codes and print operator-facing diagnostics.
- Tests inject subprocess runners instead of invoking real external CLIs.

### Integration Points
- `whilly/feedback.py` owns report body rendering and `gh issue create`.
- `whilly/cli/feedback.py` owns CLI parsing.

</code_context>

<specifics>
## Specific Ideas

The selected channel is GitHub Issues because it is the fastest path already aligned with Whilly's
repository and existing GitHub CLI integration.

</specifics>

<deferred>
## Deferred Ideas
