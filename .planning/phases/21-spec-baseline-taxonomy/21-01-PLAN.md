    Create openspec/AUTHORING.md as the canonical spec authoring conventions doc for BASE-03.
    It MUST document, sourced from 21-RESEARCH.md verified facts:
    (a) Spec file location pattern openspec/specs/<capability-slug>/spec.md, slug = directory name = spec ID, kebab-case.
    (b) The two required ## sections: `## Purpose` (>=50 characters or strict WARNING => fail) and `## Requirements` (>=1 child requirement).
    (c) Requirement block format: `### Requirement: <name>` (exactly 3 hashtags) with the normative statement on the FIRST body line containing the literal word SHALL or MUST (case-sensitive), followed by >=1 `#### Scenario:` (exactly 4 hashtags) using `- **WHEN**` / `- **THEN**` bullets and optional `- **AND**`.
    (d) A strict validation checklist enumerating every ERROR/WARNING failure mode from the research's verified table (missing Purpose/Requirements, 0 requirements, body lacks SHALL/MUST, requirement with 0 scenarios, delta header in a main spec, Purpose < 50 chars).
    (e) An explicit Anti-Patterns section that forbids delta headers (`## ADDED Requirements` / MODIFIED / REMOVED / RENAMED) in main openspec/specs/*/spec.md files (delta headers are legal ONLY under openspec/changes/*/specs/*), forbids putting SHALL/MUST only in the header, forbids bullet-list scenarios, forbids `### Scenario:` with 3 hashtags, and forbids descriptive ("does X") instead of normative ("the system SHALL X") language.
    (f) The validation command authors run: `openspec validate <slug> --strict --json`.
    Use RFC 2119 normative-language guidance. Keep lines <= 120 chars. Do NOT inline a full example spec body that itself contains `## Requirements` — to avoid a self-validating-document trap, show illustrative fragments using placeholder angle-bracket text only, never a complete second valid spec.
  </action>
