<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| operator shell → daemon args | `--issue`, `--interval` are operator-supplied; issue ref crosses into Jira HTTP |
| daemon → status/pid files | local FS writes the operator and other tooling read |
| daemon → Postgres (optional) | audit-event payloads cross into the DB when WHILLY_DATABASE_URL set |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-20-01 | Tampering | `--issue` ref reaching Jira HTTP | mitigate | `collect_jira_work_snapshot` calls `parse_jira_key` before any HTTP (verified jira_watch.py:61); loop passes the raw ref through, no new sink |
| T-20-02 | Spoofing/DoS | stale or forged PID file | mitigate | refuse-and-hint only; `os.kill(pid,0)` is liveness probe (signal 0, no delivery); NEVER send a real terminating signal to the discovered pid |
| T-20-03 | Information Disclosure | status file / audit payload | mitigate | status dict and event payload contain only pid/counts/timestamps/result; never JIRA_API_TOKEN or DSN value (reuse Phase 19 redaction discipline) |
| T-20-04 | DoS | unbounded status-file growth | accept | status file is atomically overwritten each cycle (fixed-size), not appended — no growth |
| T-20-05 | Tampering | partial status/PID write on crash | mitigate | atomic tempfile + `os.replace` (state_store.py model); BaseException cleanup unlinks temp |
</threat_model>
