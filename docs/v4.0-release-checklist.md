## How to verify

A green release smoke is the single command that proves all six SCs:

```bash
# Prereqs: Docker daemon up (testcontainers spins postgres:15-alpine)
#          .venv with `pip install -e '.[dev]'` applied

.venv/bin/python -m pytest tests/integration/test_release_smoke.py -v
```

Expected output (last line):

```
============================== 6 passed in ~25s ==============================
```

A red run blocks the tag. Triage by reading the assertion message of
the failing node ‚Äî it carries the inner pytest's stdout/stderr verbatim.

## Tagging

After this checklist lands and the release smoke is green on the final
commit of the refactoring-1 plan:

```bash
# Annotated tag ‚Äî message captures the version + the release date.
git tag -a v4.0.0 \
    -m "Whilly v4.0.0 ‚Äî distributed orchestrator (Hexagonal rewrite)" \
    -m "" \
    -m "Six PRD Success Criteria green via tests/integration/test_release_smoke.py." \
    -m "Migration guide: docs/Whilly-v4-Migration-from-v3.md" \
    -m "Architecture: docs/Whilly-v4-Architecture.md" \
    -m "Worker HTTP protocol: docs/Whilly-v4-Worker-Protocol.md"

# Push the tag to origin
git push origin v4.0.0
```

The `v4.0.0` tag is what PyPI's GitHub release workflow consumes ‚Äî
publishing the wheel happens out-of-band (manual `python -m build && twine
upload` or via the existing `.github/workflows/release.yml` if it has been
re-flowed for v4 ‚Äî verify before the push).

## Out of scope for v4.0 ‚Äî tracked for v4.1

- Per-worker bearer rotation (currently shared cluster bearer; see
  [`docs/Whilly-v4-Worker-Protocol.md`](Whilly-v4-Worker-Protocol.md#authentication))

- File-based audit log mirror of `events` table
- PRD wizard / `--init` / `--prd-wizard` (v3-only)
- `whilly --reset PLAN.json` v4 equivalent
- Removal of `whilly/cli_legacy.py`
- Cross-platform smoke (windows / macOS) ‚Äî v4 is Linux-shaped

These are *known* gaps, not regressions. The PRD ([`docs/PRD-refactoring-1.md`](PRD-refactoring-1.md))
has follow-up issues for each.
