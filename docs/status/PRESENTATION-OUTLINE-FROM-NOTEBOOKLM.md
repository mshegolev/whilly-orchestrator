Вот план презентации Whilly Orchestrator на основе данных из источников:

### Слайд 1: Что такое Whilly Orchestrator
*   **«Умный брат» Ralph-loop:** Python-реализация концепции непрерывного цикла ИИ-агентов с фокусом на глубокую аналитику и автономность [1, 2].
*   **Принцип работы:** Автоматический цикл «выбор задачи → ИИ-агент → верификация → коммит → пуш» до тех пор, пока бэклог не опустеет [2, 3].
*   **Лозунг:** «I’m helping — and I’ve read TRIZ» [1, 2].
*   **Источники задач:** Прямая интеграция с GitHub Issues, Projects v2, Jira и локальными `tasks.json` [4, 5].

### Слайд 2: Архитектура: Умный цикл (TRIZ + PRD)
*   **Pre-flight аналитика:** Использование **TRIZ-анализатора** для выявления противоречий и **Decision Gate** для отсева «мусорных» задач до расхода токенов [2, 4, 6].
*   **PRD Wizard:** Интерактивный мастер создания требований, который генерирует план работ на основе документации [7, 8].
*   **Оркестрация:** TaskManager координирует батчи задач, исключая конфликты в `key_files`, и сохраняет состояние в атомарный JSON [6, 9, 10].
*   **Исполнение:** Параллельный запуск агентов в **tmux-панелях** или изолированных **git worktrees** [7, 8, 10].

### Слайд 3: Релизы первой недели (v3.0 – v3.3)
*   **v3.0.0 (19.04.2026):** Базовый релиз — запуск continuous loop, TUI-дашборда и поддержки tmux [7, 11, 12].
*   **v3.1.0 (20.04.2026):** Запуск системы **Self-Healing** (авто-фикс NameError/ImportError/TypeError) [11, 13].
*   **v3.2.0 (22.04.2026):** Интеграция с Jira, GitHub Projects v2 и внедрение многослойного конфига `whilly.toml` [11, 14, 15].
*   **v3.3.0 (24.04.2026):** **BREAKING CHANGE**: Отключение использования worktree по умолчанию для стабильности в сложных окружениях [11, 16, 17].

### Слайд 4: Технологический стек
*   **Ядро:** Python 3.10+ (поддержка и тесты для версий 3.10, 3.11, 3.12) [18, 19].
*   **Runtime:** Использование `rich` для интерфейса, `psutil` для контроля процессов и `keyring` для безопасного хранения секретов [18, 19].
*   **Принцип минимизации:** Полный отказ от библиотеки `requests` в пользу стандартной `urllib` [18, 20].
*   **Инструментарий:** Работа через subprocess с **Claude CLI**, **OpenCode**, **gh CLI** и **tmux** [18-20].

### Слайд 5: Топ-3 сложности и решения
*   **№1 Спавн процессов на macOS:** Ошибки `EAGAIN` при параллельной работе. **Решение:** Внедрен retry-loop при запуске агентов (**PR #213**) [21-23].
*   **№2 Проблема Workspace:** Изоляция через worktree конфликтовала с абсолютными путями в `.venv`. **Решение:** В v3.3.0 опция переведена в режим opt-in (**PR #212**) [22-24].
*   **№3 Парсинг внешних CLI:** Мусор в stdout от `gh pr create` ломал логику. **Решение:** Переход на строгий regex для извлечения PR URL (**PR #204**, **#205**) [22-24].

### Слайд 6: Кейс: Self-healed release v3.2.1
*   **Проблема:** В релизе v3.2.0 был баг парсинга, из-за которого Whilly ошибочно закрыл собственный исправный PR #204 [25-27].
*   **Автономное исправление:** Система `whilly-auto-loop.sh` обнаружила инцидент, проанализировала ошибку и сама открыла PR #205 с фиксом [25-27].
*   **Итог:** Версия v3.2.1 была собрана и протестирована самим оркестратором, устранившим баг в процессе «догфудинга» [26, 27].
*   **Документация:** Детальный разбор полетов сохранен в `POSTMORTEM-PR-204.md` [28, 29].

### Слайд 7: Будущее: Whilly Forge (vNext)
*   **Концепция:** Переход от ответов на вопросы к полной генерации **Draft PR** на основе одной лишь GitHub Issue [9, 30].
*   **Пайплайн (FR-1 – FR-11):** 11 этапов трансформации задачи: от Intake (прием) и Strategy (выбор тактики) до Verify и Repair [31, 32].
*   **Модульность:** Возможность замены верификаторов и композиторов PR под специфику конкретного проекта [31-33].
*   **Цель:** Полная прозрачность процесса через Timeline-события для каждой стадии [31, 33].

### Слайд 8: Метрики проекта (Неделя 1)

*   **CI/CD:** **9 зелёных матриц** (Linux/macOS/Win × 3.10/3.11/3.12) на каждый Pull Request [18, 19, 34].
